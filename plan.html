一项目概述

1.1 项目背景

建筑工程事业部现有总经理 1 人、直营主管（管 17 名大客户经理）、渠道主管（管 29 名工程专员）、 运营主管（管 9 名商务专员），合计 57 人。事业部当前面临的核心问题是：业务数据分散在公司 CRM 系统（与看板系统不兼容）、企业微信群和线下台账中，管理层无法实时掌握团队的整体业绩 进度、客户开发情况以及各层级人员的执行状态，亟需一个统一的管理全景可视化看板。

1.2 项目目标

🎯 目标一：数据同频

将散落在 CRM、企业微信、台账中的数据统一汇集，实现事业部全员查看一致的经营数据。

🎯 目标二：识别优劣

通过红黄绿灯预警和进度追踪，快速定位优秀团队与落后团队，为管理干预提供依据。

🎯 目标三：辅助决策

通过指标联动分析和趋势对比，帮助管理层推断业务走势、提前预警风险、制定改善策略。

2.7 指标联动关系图

六大指标相互影响，形成管理闭环（虚线箭头表示间接影响，实线箭头表示直接影响）：

📈 销量业绩 ⟶ 🤝 客户拜访 ↔ 👥 客户开发 ⟶ 🏆 品牌入围 - - → 🌱 代理培育 ⟶ 📚 培训赋能 - - → 📈 销量业绩

解读：

  • 销量业绩 是结果指标，受 客户开发 + 客户拜访 直接驱动
  • 客户拜访 频率和质量影响 客户开发 和 品牌入围 的推进速度
  • 品牌入围 为 代理培育 提供品牌背书，间接促进销量提升
  • 培训赋能 提升团队能力，间接影响拜访质量、客户开发和代理培育效果
  • 代理培育 是中期产能，对销量业绩产生滞后正向影响

三数据架构方案

3.1 数据现状

🔴 现状问题
  • 业务数据存储于公司 CRM 系统
  • CRM 与看板系统 不兼容，无法直接 API 对接
  • 部分数据散落于企业微信群和线下 Excel 台账
  • 缺少统一的数据汇集与展示平台
🟢 解决思路
  • 以企业微信 + 台账文件作为数据桥接层
  • 建立标准化台账模板（Excel/CSV）
  • 通过企业微信机器人实现文件上传→自动解析
  • 解析后数据自动更新前端 JSON 数据文件

3.2 数据流转方案

CRM 系统 → 手动导出 → 企业微信
发送台账 → 自动解析
(脚本) → 更新 JSON
数据文件 → GitHub Actions
自动部署 → 看板展示

3.3 实时化实现路径（分三阶段）

📋 第一期 · 手动 / 半自动
  • 专人定时从 CRM 导出数据
  • 按模板整理为 Excel 台账
  • 企业微信群提交文件
  • 手动触发更新
  • 更新频率：日报 / 月报
⚙️ 第二期 · API 对接
  • 推动 CRM 开放数据接口
  • 或搭建中间数据仓库
  • 减少人工导出环节
  • 增加数据校验自动化
  • 更新频率：日报自动
🚀 第三期 · 实时同步
  • 实时 / 准实时数据同步
  • 看板自动刷新（30秒轮询）
  • 异常数据自动告警
  • 数据血缘追踪
  • 更新频率：准实时

3.4 推荐方案：企业微信机器人 + 台账模板

环节方案说明负责人
台账模板统一 Excel/CSV 模板，含必填字段、数据校验、下拉选择运营主管

4.2 卡片设计规范

📐 卡片结构（每个板块）
  • 顶部：指标名称 + 红黄绿灯状态标识
  • 中间主区域：环形进度图 / 柱状图，展示完成率
  • 下方：实际值 / 目标值 / 考核线 三组数据
  • 角标：同比 / 环比变化指示
🎨 色彩规范
  • 绿灯 完成 ≥ 阈值上线，正常推进
  • 黄灯 完成在阈值区间，需关注
  • 红灯 完成低于阈值下线，需立即干预
  • 主色：深蓝 #0a2a5c / 白色 #fff
📊 图表类型
  • 完成率：环形进度图（doughnut）
  • 趋势：折线图（月度 / 季度走势）
  • 对比：柱状图（团队间横向对比）
  • 使用 Chart.js 轻量渲染

4.3 红黄绿灯预警展示方式

状态条件看板展示管理动作
绿灯 完成率 ≥ 阈值上线 绿色圆点 + 正常色进度条 维持现状，分享优秀经验
黄灯 阈值下线 ≤ 完成率 < 阈值上线 黄色圆点 + 黄色进度条 + 警示图标 主管跟进，分析原因，制定改善计划
红灯 完成率 < 阈值下线 红色圆点 + 红色进度条 + 闪烁动画 + 弹窗提醒 总经理介入，专项会议，资源倾斜

4.4 移动端与 PC 端适配方案

📱 移动端（手机）
  • 单列布局，卡片纵向排列
  • 顶部固定导航 + 底部操作栏
  • 卡片可滑动切换
  • 红黄绿灯状态清晰可见
  • 支持横屏查看图表详情
💻 PC 端（桌面）
  • 2×3 或 3×2 网格布局
  • 左侧固定组织架构树（后续）
  • 右侧详情面板（点击展开）
  • 多维度筛选（时间 / 团队 / 指标）
  • 支持大屏投屏展示
📟 适配策略
  • CSS Grid / Flexbox 自适应
  • 使用 rem / vw 相对单位
  • 媒体查询断点：768px / 1024px
  • 触控与鼠标事件双重支持
  • 字体大小随屏幕自动调整

4.5 交互设计

🔽 下钻功能

点击任意指标板块 → 进入详情页，展示该指标的完整数据（团队成员排名、历史趋势、明细列表）。

📅 时间筛选

顶部提供「月度 / 季度 / 年度」切换按钮，所有卡片同步更新对应时间维度的数据。

📥 数据导出

看板数据支持导出为 Excel / CSV / 截图，一键下载，方便周报和会议汇报使用。

🔔 预警推送

当指标进入黄灯/红灯区间时，看板自动推送企业微信提醒，并高亮显示风险项。

5.3 更新机制

🔄 数据更新流程
  1. 运营人员整理台账 → 发送至企业微信群
  2. 企业微信机器人接收文件 → 触发解析脚本
  3. 脚本解析 Excel/CSV → 生成 JSON 数据文件
  4. 脚本自动 commit & push 到 GitHub 仓库
  5. GitHub Actions 检测到推送 → 自动部署到 Pages
  6. 看板自动刷新展示最新数据

七风险与应对

项目实施过程中可能面临以下风险，提前识别并制定应对策略：

高风险
📌 CRM 数据无法导出
公司 CRM 系统存在数据导出限制，或导出格式不满足台账模板要求。
应对：提前协调 IT 部门确认导出权限；若受限则采用人工录入方式作为兜底方案。
高风险
📌 数据更新不及时
运营人员未按时上传台账，导致看板数据滞后，失去参考价值。
应对：设置企业微信定时提醒；建立数据更新 SOP 并纳入绩效考核；设置超时未更新自动告警。
中风险
📌 数据准确性问题
台账数据填写错误或格式不一致，导致看板展示数据失真。
应对：台账模板增加数据校验规则；解析脚本增加异常检测（数值范围、完整性检查）；运营主管复核机制。
中风险
📌 用户接受度不足
团队成员习惯原有方式，不主动使用看板，项目价值无法体现。
应对：组织看板使用培训；管理层带头使用；看板内容与周会/月会挂钩；收集反馈持续优化。
低风险
📌 网络与安全风险
GitHub Pages 在国内访问稳定性，以及业务数据在外网传输的安全性。
应对：考虑 GitHub Pages + 国内 CDN 加速；敏感数据脱敏后再上传；后续可迁移至企业内网服务器。
低风险
📌 人员变动影响
负责台账维护或脚本开发的人员变动，导致流程中断。
应对：关键文档与脚本进行版本管理（Git）；至少 2 人掌握核心流程；定期复盘流程执行情况。
